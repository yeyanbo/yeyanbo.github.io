3.3 FHIR文件

FHIR基础设施工作组成熟度等级：3标准状态： 试用

FHIR资源可用于构建代表组合的文档：一组连贯的关联信息，是医疗保健信息的陈述，包括临床观察和服务。文档是一组由人、组织和设备创建、证实的不可变的资源的固定表达形式。

以这种方式构建的文档可以在系统之间交换并保存在文档存储和管理系统中，包括像IHE XDS之类的系统。

声称符合此框架的申请声称符合“FHIR文件”（参见符合性）。

FHIR文件可能是“临床”（专注于患者医疗保健信息），但也可能用于非临床目的（例如FHIR实施指南，实践指南，患者讲义等）.HL7将在未来制定概况，为适当的代表提供额外指导一般临床文件以及特定类型的临床文件（例如综合CDA）。

请注意，FHIR定义了此文档格式和文档引用资源。FHIR文档适用于在FHIR中创作和汇编的文档，而文档参考资源用于对预先存在的文档进行一般性引用。

  • 出院摘要示例：XML或JSON

3.3.1文件内容

所有文档都具有相同的结构：“文档” 类型 的资源包，其中包含Composition资源作为包中的第一个资源，后跟一系列其他资源，从Composition 资源引用，为文档提供支持证据。该捆绑包将文档的所有内容收集到单个XML或JSON文档中，可以根据需要对其进行签名和管理。资源包括人类可读和计算机可处理的部分。此外，捆绑包可能包括CSS样式表， 原产地声明和签名。

组成资源是临床文件的基础。它：

  • 提供身份及其目的，并设置文档的上下文
  • 包含主题和作者等关键信息，以及证明该文档的人员
  • 将文档分成一系列部分，每个部分都有自己的叙述

组装文档时，下面列出的组合引用的资源应包含在软件包中：

  • Composition.subject
  • Composition.encounter
  • Composition.author
  • Composition.attester.party
  • Composition.custodian
  • Composition.event.detail
  • Composition.section.author
  • Composition.section.focus
  • Composition.section.entry

如果文档构造系统选择这样做，则这些引用的资源所引用的其他资源也可以包括在捆绑中。包括这些额外的资源将使文档更大，但如果在处理文档时需要检索链接的资源，则可以节省应用程序。因此，是否应包括这些链接的资源取决于实施环境。

文档包应仅包括：

  1. Composition资源，以及直接或间接（例如递归地）从中引用的任何资源
  2. 包含样式表的二进制资源（如下所述）
  3. 原型资源具有组合目标或文档中包含的其他资源

文档中有两个关键标识符：

  • 文档标识符（必填）。这可以在Bundle.identifier中找到，并且对于此文档实例是全局唯一的，并且永远不会被重复使用，包括从相同组合派生的其他文档
  • 组合标识符（可选）。这可以在Composition.identifier中找到，并且对于从此组合派生的所有文档都是相同的

该文件有几个日期：

  • 文件日期（强制性）。这可以在Bundle.meta.lastUpdated中找到，并确定何时从底层资源组装文档包
  • 作文日期（必填）。这可以在Composition.date中找到，这是作者在逻辑上编写文档的时候
  • 证明日期（可选）。这可以在Composition.attester.time中找到，并且是证明者见证文档的时间。这通常与组成日期或之后同时进行
  • 作文最后修改时间（可选）。这可以在Composition.meta.lastUpdated中找到，并且是组合物的最后更改日期。这必须> =组成日期

可以按照数字签名页面中规定的规则使用数字签名对文档包进行签名。签名应该由列出的文档提供者提供，签名应该包含一个KeyInfo元素，该元素包含一个KeyName元素，该元素的值是与匹配的证明者资源的fullUri匹配的URI 。

一旦组装成一个包，文档就是不可变的 - 它的内容永远不会被改变，文档id永远不能被重用。请注意，文档可以用XML或JSON表示，并在这些文档之间进行相互转换或更改其字符编码，同时保持相同的文档。然而，文档中直接引用的内容和文档的呈现不能显着改变（使得它改变内容的临床意义）。从相同组合派生的任何其他文档都应具有不同的文档ID。

3.3.1.1文件介绍

当文件被呈现供人类消费时，应用程序应该按顺序呈现整理的叙述部分：

  1. 该主题资源叙事
  2. 该组合资源叙事
  3. 该section.text叙事

该文档的呈现称为文档的“证明内容”。附加资源可以包含在捆绑中（例如，从List中引用的资源，表示section.content应该在捆绑中，并且可以包括它们引用的其他资源），但是这些（以及任何叙述）都不是证明的内容。具体而言，Composition.attester证明了文件的呈现形式。

组合资源叙述应总结文档标题的重要部分，这些部分是为文档建立临床上下文所必需的（除了主题，它本身就显示）。要真正构建组合叙事，只需将所有叙述性<div>片段附加在一起即可。

如果文档的显示顺序与上面给出的顺序不同，则可能不代表原始证明的内容。实施指南可能会进一步限制文档叙述和显示行为。

该XML工具的参考实现包括XSLT转换一种转换XML文档转换成浏览器的准备XHTML。

除了必须遵循的关于叙事的基本样式规则之外，文档还可以引用或包含一个或多个样式表，其中包含适用于整理叙述的其他样式。这是通过在Feed上声明样式表链接来完成的： 

<Bundle xmlns =“http://hl7.org/fhir”> 
  <！ - 元数据和类型 - > 
  <link> 
    <relation value =“stylesheet”/> 
    <url value =“[uri]”/> 
  < / link> 
</ Bundle>

它url可以是对CSS样式表的绝对引用，也可以是对带有CSS样式表的二进制资源的相对引用。样式表引用只能引用CSS样式表 - 其他形式的样式表是不可接受的。

相对（内部）引用应该用于样式表，因为由于技术问题或本地策略决策，查看者可能无法在查看时解析外部内容。

引用或使用的任何样式表不得以改变内容临床含义的方式改变演示文稿。

除非在当地贸易伙伴协议中另有约定，否则显示整理叙述的应用程序应该使用文档指定的样式表（请参阅安全说明）。签订贸易协议的缔约方应非常认真地考虑该行动对其长期文件交换范围的影响。如果双方同意使用未包含在文档中的样式表，则可能是他们永远无法在更一般的环境中安全地共享其文档，例如区域或国家EHR或全球个人健康记录。

3.3.2文档配置文件

文档配置文件用于描述特定用途的文档。文档配置文件可以制定以下规则

  • Composition资源的内容，包括
  • 组成部分的结构
  • 捆绑包中包含哪些资源以及Document资源中直接引用的资源

应用程序应考虑发布标识其支持的文档类型的功能语句。文档可以通过在Bundle.meta.profile 元素中放置配置文件标识符来标识它们符合的配置文件- 有关此实用程序的讨论，请参阅配置文件标记。

3.3.3文件处理义务

临床文件的作者/构造者和处理者，无论是人类还是软件，都有他们必须满足的义务。

3.3.3.1 作者/构造者义务

文档构造函数是创建文档的应用程序。作者是使用构造函数创建文档的人，组织或设备。在它们之间，构造器和作者可以在执行其任务时创建新的内容资源和/或组合已经存在的内容资源。他们还有以下责任：

  • 确保文档应包含符合此处描述的规则且仅链接到其他有效资源的有效组合
  • 确保文档内容符合任何声明的配置文件（见下文）。
  • 确保证明者正确地了解他们所证明的文件的陈述

3.3.3.2 处理器义务

文档处理器是应用程序和/或人类用户，其接收文档并从中提取数据或由于它们而做出决定。文档可以直接从文档构造器接收，通过文档管理系统访问或由第三方转发。文档处理器负责确保根据本规范处理和/或呈现所接收的文档。文档处理者有义务确保遵循以下规则：

  • 在存储/传输文档时，可以使用任何方法，只要捆绑的文档可以（重新）组装，具有足够的完整性以验证数字签名。
  • 在介绍文件的叙述时，应遵循上述规则
  • 可以提取来自文档的资源或数据以用于其他用途，但是文档作者不再认为这些数据被证明
  • 无论将文档中的数据显示给用户，应该始终为用户提供访问原始文档的表示的方式。

除了这些义务之外，文件接收者还应该仔细跟踪取代现有文件的新文件的文件来源，特别是当文件代表已经收回的文件时。更换文件后，他们应该撤回从被取代文件中提取的数据，或者在用户查看文件或从中取出数据时发出警告。

3.3.4文件终点

处理文档时使用了几个不同的RESTful端点。通过考虑发布给他们的后果，可以最好地描述各种终点的使用：

终点 内容类型 描述
[baseURL时] /捆绑 文件包 这类似于管理资源类型的正常终点，但它适用于整个文档包 - 即读取操作返回包，更新获取包，搜索返回捆绑包。请注意，如果文档使用创建交互进行POST，则Bundle.id将更改，但Bundle.identifier不会。请参阅使用RESTful API提供服务包以获取进一步的评论
[baseURL时] /排版 作文资源 管理组合资源的正常终点。这可以在构建文档时或在将文档分解为其组成资源之后或在与文档分开使用组合时使用
[baseURL时] /二进制 文件包 只需将整个文档存储为一个字节序列，并在请求时返回该序列。无法在/ Binary端点中查找内容，因此通常会将其与文档参考相关联，以便应用程序可以查找和处理文档，但这不是必需的
[baseurl]（例如交易） 文件包 忽略捆绑包是文档并将其包含的所有资源作为单独资源进行处理的事实。客户端不应该期望接收使用此方法提交的文档的服务器能够准确地重新组合文档。（即使服务器可以重新组合文档（见下文），也不能期望结果具有相同的顺序等。因此，文档签名很可能无效。）。有关更多详细信息，请参阅接受其他Bundle类型

注意：虽然这些端点定义为与文档相关的资源和文档包一起使用，但没有必要使用它们。可以使用任何所需的方法在系统之间传输文档。此外，服务器和/或规范可以定义 用于处理超出上述选项的文档的附加操作。

3.3.4.1 生成文档

客户端可以要求服务器从合成资源生成完全捆绑的文档。有关详细信息，请参阅生成文档操作。
